# 论文筛选二分类多路径算法 A/B 需求

## Problem Frame

当前已经具备一套约 1000 条、经过人工与 AI 标注的论文筛选数据集，以及可用的评测系统。现有主仓评测 API 以单篇论文判别为主，对外输出 `positive/negative`，并在命中正例时附带一个主偏好标签。

这次目标不是泛化地讨论“推荐算法”，而是围绕现有代码和现有评测契约，先把“大类二分类”任务做成一个稳定样板：先判断一篇论文是否属于用户关心的偏好论文，即 `positive/negative`。子类标签暂不作为第一阶段主目标，后续再在稳定二分类之上继续细分。

用户希望：

- 同时保留多条算法路线并行推进
- 通过 A/B 测试择优录取
- 第一阶段优先保障召回率
- 尽量少依赖人工规则
- 可以把外部模型 API 作为正式候选算法纳入评测

此外，当前已知单个子类如 `解码策略优化` 的正样本仍较少，这说明如果过早把目标收缩到子类级别，会显著放大小样本风险。因此第一阶段先做大类二分类，更符合当前数据现实。

## Requirements

- R1. 第一阶段只聚焦大类二分类任务，评测口径按 `positive/negative` 统计 precision 与 recall。
- R2. 目标指标为 `recall > 90%` 且 `precision > 50%`，两者缺一不可。
- R3. 第一阶段必须采用多路径并行方案，而不是只实现单一路线。
- R4. 所有候选方案必须能接入现有评测系统，输出与现有二分类评测契约兼容的结果。
- R5. 候选方案允许包含外部模型 API，也允许包含本地模型，但必须在同一评测集上做统一对比。
- R6. 第一阶段优化优先级为“先保召回，再逐步抬 precision”，不接受为了高 precision 明显牺牲 recall 的方案作为主推荐路线。
- R7. 第一阶段不应以大规模手工规则工程作为主方案，只允许少量规则作为召回兜底、候选生成或安全护栏。
- R8. 第一阶段至少覆盖三类候选路线：
  - 学习式本地分类路线
  - embedding / 相似度检索路线
  - LLM / 外部 API 判别路线
- R9. 除单模型路线外，必须至少评估一种两阶段路线：
  - 高召回候选生成
  - 高精度裁决
- R10. A/B 结果需要能回答：
  - 哪条路线 recall 最稳
  - 哪条路线 precision / recall 综合最好
  - 哪条路线对小样本最鲁棒
  - 哪条路线最值得作为后续子类细分的底座
- R11. 第一阶段方案设计应显式考虑小样本条件，不默认假设可以依赖大规模监督训练获得稳定效果。
- R12. 第一阶段方案应支持后续扩展到子类细分任务，但本次不要求同时把六个偏好标签都建模或都达标。
- R13. 多条候选路线必须支持在仓库内并行开发，彼此尽量减少代码与分支冲突。
- R14. 并行开发默认采用 Git worktree 隔离：共享底座与各路线实验应分开演进，而不是多人反复在同一工作目录切分支。
- R15. A/B 框架必须提供统一输入输出契约，使各路线可以在独立 worktree 中实现，但仍能被同一 runner 拉起和比较。

## Success Criteria

- 至少形成 3 条可以真实评测的候选算法路线，而不是停留在概念层比较。
- 大类二分类任务上至少出现 1 条候选路线达到或逼近 `recall > 90%`、`precision > 50%`。
- 团队可以基于同一套离线结果清晰判断：
  - 继续深挖哪条路线
  - 哪些路线应淘汰
  - 哪些组件值得复用到后续子类任务
- 第一阶段结论能够沉淀为后续子类细分的统一方法模板，而不是一次性的特例实现。
- 候选路线可以在不同 worktree 中独立推进，并最终在统一 A/B 框架下直接比较，而不是先相互手工合并后才能评测。

## Scope Boundaries

- 本阶段不要求六个偏好标签同时都达到目标指标。
- 本阶段不要求第一步就解决子类判别问题。
- 本阶段不要求先解决完整多标签推荐或全量排序问题。
- 本阶段不要求去改造公共评测协议的对外输出约束。
- 本阶段不以“纯规则系统”作为主方向。
- 本阶段不先追求线上实时部署成本最优，优先追求离线评测优胜。
- 本阶段不鼓励多条候选路线长期共用同一实现文件反复改写；应优先通过 worktree 和模块边界降低互相影响。

## Key Decisions

- 先做大类二分类，不先做子类：
  因为当前单个子类正样本较少，若一开始直接攻子类，容易让评测被小样本噪声主导；先把 `positive/negative` 做稳，更符合当前数据规模与业务节奏。
- 多路径并行而不是预设赢家：
  因为当前数据规模仍有限，单看经验很容易误判，必须让本地学习器、embedding 路线和 LLM 路线在统一评测下竞争。
- 优先保障 recall：
  因为当前业务目标是不要漏掉真正相关论文，precision 可以通过第二阶段裁决、阈值调优和 hard negative 扩充逐步提升。
- 少规则约束保留，但允许最小兜底规则：
  完全零规则在小样本高召回场景下风险过高，因此允许少量高价值规则作为护栏，但不把规则工程作为主体。
- 两阶段路线必须参赛：
  在小样本场景下，单段式分类器往往难同时兼顾高 recall 和可接受 precision，而“候选生成 + 裁决”更有机会过线。
- 并行开发应以 worktree 为默认协作方式：
  因为多条候选路线需要并行推进，若都直接改同一份实现，很容易造成分支冲突、实验互相污染，以及难以回溯每条路线的真实效果。
- 共享底座与路线实现应明确分层：
  公共协议、数据读取、A/B runner、结果落盘格式应尽量集中到共享底座；具体路线只改自己的实现目录和少量注册点，避免候选方案互相侵入。

## Alternatives Considered

- 只做纯关键词 / 规则系统：
  预计启动快，但不符合“尽量少规则”的方向，也容易在标签扩展时失去可维护性。
- 直接做单一监督分类器：
  在正样本少于 50 条时风险较高，容易因样本不足导致 recall 不稳。
- 直接把外部 LLM 作为唯一主链路：
  虽然可能快速取得较高上限，但缺少与本地方案对照时，很难判断是否具备成本与稳定性优势。

## Dependencies / Assumptions

- 现有数据集和评测系统已经可用。
- 主仓现有评测 API 对外维持 `positive/negative` 判别输出，并允许 positive 时附带一个主偏好标签。
- 当前部分子类如 `解码策略优化` 的正样本量少于 50。
- 第一阶段允许使用外部模型 API 做正式对照实验。
- 仓库协作允许使用 Git worktree 管理多条候选路线。

## Outstanding Questions

### Deferred to Planning

- [Affects R3][Technical] 候选路线的最小实现集合应包含哪些具体算法组合，才能覆盖“本地学习器 / embedding / LLM / 两阶段”四类能力。
- [Affects R4][Technical] 现有评测入口应如何封装成统一 A/B runner，以便同一论文集重复跑多条路线。
- [Affects R6][Needs research] 在当前数据规模下，召回优先的阈值搜索和校准策略应该如何设计，才能避免 precision 被过度拉低。
- [Affects R7][Technical] 允许保留的“最小规则兜底”边界是什么，哪些规则属于护栏，哪些规则已经变成主逻辑。
- [Affects R8][Needs research] embedding 路线更适合作为独立分类器，还是更适合作为候选召回器。
- [Affects R8][Needs research] 外部 LLM 判别路线更适合作为单阶段判别器，还是更适合作为二阶段裁决器。
- [Affects R10][Technical] A/B 报表除 precision / recall 外，是否还需要成本、延迟、可解释性和稳定性维度。
- [Affects R12][Needs research] 从大类二分类迁移到后续子类任务时，哪些组件可以共享，哪些部分必须标签专属化。
- [Affects R13][Technical] worktree 并行开发时，哪些文件属于共享底座、哪些文件应由路线独占，才能最大限度减少冲突。
- [Affects R14][Technical] 每条候选路线的 worktree / 分支命名、目录布局和合并策略应如何约定，才能既方便实验又不污染主线。
- [Affects R15][Technical] 统一 A/B runner 的最小接口应如何设计，才能让不同 worktree 中的路线实现无需互相依赖也能被统一拉起。
