# 论文筛选评测双标签体系草案

## Problem Frame

当前目标不是先做“大而全”的 AI 论文分类，而是先建立一套能服务用户真实筛选需求的离线评测体系。

用户关注的核心偏好并不是整个 AI 学术版图，而是少数高价值方向，当前明确包括：

- 推理优化
  - 第一版正式子类已确定
- Agent
  - 例如上下文管理、记忆召回、多 Agent 编排调度等
- LLM / VLM 模型算法
  - 例如线性 Attention、Diffusion LLM 等

在讨论后，决定第一版不再追求统一的大 taxonomy，而改为建立两套相对正交的标签体系：

- 偏好标签体系
  - 用于直接表达用户真正关心的筛选偏好，第一版先从 `推理优化` 及其子类开始
- 研究对象标签体系
  - 用于表达论文主要研究什么对象，第一版采用中粒度分类，例如 `LLM`、`多模态`、`Diffusion`、`机器学习` 等

这样做的目标是：

- 让 benchmark 直接服务筛选算法优化
- 让标注成本保持可控
- 让后续组合查询成为可能
  - 例如“找 `LLM` 里的 `推理优化` 论文”

## Requirements

- R1. 第一版 benchmark 以“论文 + 双标签标注”作为基础结构。
- R2. 双标签结构必须至少包含两条轴：
  - A 轴：偏好标签体系
  - B 轴：研究对象标签体系
- R3. A 轴第一版仅聚焦 `推理优化` 及其子类，不要求同时覆盖 Agent 和模型算法全部偏好。
- R4. B 轴第一版采用中粒度研究对象分类，而不是粗粒度或全领域细粒度分类。
- R5. 每篇论文在 B 轴上必须有一个主研究对象标签。
- R6. 每篇论文在 A 轴上允许有零个、一个或多个偏好标签。
- R7. 第一版离线评测优先支持二分类与集合筛选评测：
  - 对给定偏好标签，判断论文是否相关
  - 计算 precision / recall
- R7a. 第一版离线评测必须显式定义负样本策略，而不是仅以“未命中标签”笼统作为负例。
- R7b. 对每个 A 轴偏好子类，测试集负样本至少应分为三层：
  - easy negatives：随机负样本
  - in-domain negatives：同 B 轴主类但未命中目标 A 轴标签的负样本
  - hard negatives：与目标标签语义接近、但经人工确认不属于目标标签的负样本
- R7c. 评测报告中应分别统计不同负样本层上的表现，避免单一总体指标掩盖真实筛选难度。
- R8. 第一版评测结果需要支持组合分析：
  - 按偏好标签统计
  - 按研究对象标签统计
  - 按“研究对象 × 偏好标签”交叉统计
- R9. 研究对象标签体系应参考业界主流做法，但必须服从用户实际筛选需求，不以学术完备性为首要目标。
- R10. 第一版 benchmark 的标签设计应以本地 `paperlists` 可见的 2025 到 2026 顶会数据为参考，优先参考：
  - `AAAI 2025`
  - `ICLR 2025`
  - `ICLR 2026`
  - `ICML 2025`
  - `NeurIPS 2025`
- R11. 如果标签体系复杂度过高，会增加标注与算法实现难度，因此第一版标签数量必须有明确上限并保持可解释。

## Success Criteria

- 用户关注的第一类核心偏好“推理优化”可以被拆成一组稳定、可标注、可计算指标的子标签。
- 标注员可以对大多数论文稳定给出：
  - 一个研究对象主标签
  - 若干偏好标签
- 可以对如下问题进行离线评测：
  - “推理优化论文是否被召回”
  - “LLM 中的推理优化论文是否被召回”
  - “多模态中的推理优化论文是否被错误混入”
- 可以区分模型是在“排除明显无关论文”上表现良好，还是在“区分相近论文边界”上也足够稳定。
- 第一版 benchmark 的规模和标签复杂度足以支持算法迭代，但不会因为 taxonomy 过大而拖慢推进。

## Scope Boundaries

- 本阶段不做全 AI 顶会统一 ontology。
- 本阶段不要求 Agent、LLM/VLM 模型算法两大偏好簇在第一版全部纳入 benchmark。
- 本阶段不定义完整的开放自然语言偏好解析系统。
- 本阶段不要求对每篇论文做无限细的多层主题拆解。
- 本阶段优先解决离线评测可行性，而不是一次性解决所有检索表达。

## Key Decisions

- 第一版标签体系改为“双轴”而不是“一棵大树”：
  这样可以把“用户偏好”和“研究对象”分开建模，避免单一 taxonomy 同时承载过多语义。
- 第一版偏好轴只做 `推理优化`：
  因为这是当前最明确、最想优化、也最适合先建立评测闭环的方向。
- 第一版 `推理优化` 正式子类固定为 6 个：
  先收敛为稳定父类，避免从一开始就把 benchmark 做成过细的叶子标签表。
- 第一版 A 轴允许一篇论文同时命中多个 `推理优化` 子类：
  这更符合真实论文跨方法、跨优化层次的情况，避免为了单标签而扭曲标注。
- 第一版研究对象轴采用中粒度：
  这样既能支撑组合筛选，也不会像全领域 taxonomy 那样过重。
- 业界 taxonomy 仅作为参考骨架，而不是直接照搬：
  因为产品目标是服务你的筛选需求，而不是构建通用学科分类系统。

## Dependencies / Assumptions

- 本地 `paperlists` 可用于参考的 2025-2026 会议信息包括：
  - `AAAI 2025`
  - `ICLR 2025`
  - `ICLR 2026`
  - `ICML 2025`

- `系统与调度优化`
- `算子与内核优化`
- `模型结构侧推理优化`

#### A2 子类解释与示例边界

- `解码策略优化`
  - 例如投机推理、并行解码、early exit、adaptive computation、test-time scaling
- `上下文与缓存优化`
  - 例如 KV Cache、上下文压缩、长序列 Token 压缩、context pruning、memory selection
- `系统与调度优化`
  - 例如弹性调度、动态 batching、request scheduling、prefill / decode 解耦
- `算子与内核优化`
  - 例如 attention kernel 优化、kernel fusion、编译优化、算子级 memory / bandwidth 优化
- `模型结构侧推理优化`
  - 例如线性 Attention、MoE 推理路由优化、state space / hybrid efficient inference architecture
- `模型压缩`
  - 例如量化、蒸馏、剪枝、低比特推理、压缩感知部署

#### A 轴设计约束

- 第一版 benchmark 正式启用以上 6 个子类。
- 第一版暂不强制继续向更细叶子标签拆分。
- 如果后续某一子类内部论文数量过大，再单独扩展到下一层叶子标签。
- 第一版允许多标签命中，不强制单主子类。

- 当论文跨多个研究对象时，按“主要贡献所服务的对象”决定主类。
- 如果论文本质是 benchmark、数据集或 evaluator，可优先归入 `评测 / Benchmark / 数据集`。

## Industry Alignment

- 参考业界主流做法，研究对象标签应保持相对稳定，类似“主 topic”。
- 偏好标签更接近 task / method / optimization target，应允许更贴近业务目标，而不必与学科分类完全一致。
- 这与 OpenAlex 的“主 topic + 多 topic”思路、以及 Papers with Code 的“任务 / 方法 / 数据集”思路是兼容的。

## Recommendations

- 第一版 benchmark 直接以这 6 个子类作为正式偏好标签。
- 每篇论文可命中：
  - 零个子类
  - 一个主偏好子类
  - 必要时多个偏好子类
- 后续如发现多标签冲突频繁，可增加“主偏好子类 + 次偏好子类”约束。

## Negative Sampling Strategy

### 定义

负样本策略指的是：在评测某个 A 轴偏好标签时，如何系统地选择“不属于该标签”的论文作为反例，用来检验筛选算法是否真正学会了边界，而不只是排除明显无关样本。

在本 benchmark 中，负样本策略不是可选细节，而是评测协议的一部分。因为如果负样本过于简单，precision / recall 往往会虚高，无法真实反映算法在相似论文间的区分能力。

### 为什么需要分层负样本

- 仅使用随机负样本时，评测更像“能否排除明显无关论文”，难度偏低。
- 用户真实筛选场景中，干扰项通常来自同一研究对象内的相似论文，而不是全领域随机论文。
- 论文筛选系统真正困难的部分，往往是区分“很像目标标签、但核心贡献不属于目标标签”的边界样本。

因此第一版 benchmark 采用三层负样本结构，而不是单一负样本池。

### 三层负样本

#### 1. easy negatives

- 从全体论文池中随机抽取未命中目标 A 轴标签的论文。
- 目的：
  - 检验算法是否具备基础的相关 / 不相关区分能力
  - 提供低成本、低噪声的基线评测
- 风险：
  - 难度通常偏低
  - 不能代表真实筛选时遇到的主要干扰项

#### 2. in-domain negatives

- 从与正样本相同 B 轴主类的论文中抽取未命中目标 A 轴标签的论文。
- 例如评测 `LLM × 上下文与缓存优化` 时：
  - 正样本是 `LLM` 中命中该 A 标签的论文
  - in-domain negatives 是其他 `LLM` 论文，但不属于该 A 标签
- 目的：
  - 模拟更真实的候选集分布
  - 检验算法能否在相同研究对象内部做细粒度区分

#### 3. hard negatives

- 与目标标签在标题、摘要、关键词或技术表述上高度相似，但按标注规则明确不属于目标 A 标签的论文。
- 这类样本需要人工确认，不能完全依赖自动抽样。
- 常见来源包括：
  - 标题中出现 `efficient`、`fast`、`scaling` 等词，但核心贡献不是推理优化
  - 做 benchmark、数据集、评测框架，但不是优化方法本身
  - 属于相邻 A 子类，容易被误判为目标子类
- 目的：
  - 检验算法是否真正学会标签边界
  - 暴露“语义相似但不相关”的误召回问题

### 与双轴标签体系的关系

- A 轴决定“当前评测的目标标签是什么”。
- B 轴决定“in-domain negatives 如何构造”。
- `研究对象 × 偏好标签` 的交叉分析，不仅用于报表，也直接用于负样本分层。

换句话说，双轴结构不仅服务统计分析，也服务评测集构筑。

### 第一版推荐构造方式

对每个 A 轴子类，按以下方式构造测试集：

- 正样本：
  - 命中该 A 轴子类的论文
- easy negatives：
  - 从未命中该子类的全局论文池中随机抽样
- in-domain negatives：
  - 从相同 B 轴主类、但未命中该子类的论文中抽样
- hard negatives：
  - 从与目标标签语义接近的候选池中召回，再由人工复核确认

第一版不要求每个子类的三层负样本规模完全相同，但要求：

- 每个子类都至少有三层负样本
- hard negatives 的比例不能为 0
- 报告中需要区分不同层的结果

### 报表要求

对每个 A 轴子类，至少输出：

- overall precision / recall
- easy negatives 上的 precision / recall
- in-domain negatives 上的 precision / recall
- hard negatives 上的 precision / recall
- 按 B 轴主类分桶的结果
- `B × A` 交叉统计中的主要误召回分布

这样可以回答以下问题：

- 模型是否只会排除明显无关论文
- 模型在同一研究对象内部是否仍能稳定识别目标偏好
- 模型最容易把哪些“相邻但不相同”的论文误判为相关

### 第一版实施建议

- 先以 easy negatives + in-domain negatives 建立最小可运行版本。
- hard negatives 可以先从误判样本和相邻子类中半自动采样，再做人审。
- 当某个 A 子类的 hard negatives 数量明显不足时，优先补这部分，而不是继续扩张标签体系。
- 如果后续需要训练集与测试集分离，负样本策略也应在不同 split 中保持协议一致，避免训练和测试难度结构不一致。
