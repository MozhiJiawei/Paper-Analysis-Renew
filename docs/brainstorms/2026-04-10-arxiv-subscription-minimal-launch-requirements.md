# arXiv 订阅最小上线

## Problem Frame

当前仓库已经具备真实 arXiv 订阅抓取、报告产物写出和本地静态 HTML 渲染基础，但“真实订阅 -> 推荐 -> 定时运行 -> 邮件送达 -> 本地可查看报告”的上线闭环还没有打通。

这次目标不是扩展新的产品面，而是以 Agent 为第一用户，尽快把真实 arXiv 订阅链路上线，让系统每天自动运行，并把推荐结果稳定送达到用户邮箱，同时在本地保留一个可扩展的 HTML 报告页面，作为后续用户反馈与入库能力的承载壳子。

## Requirements

- R1. 系统必须支持被外部调度器触发后，自动执行真实 arXiv 订阅抓取和当前推荐算法处理链路。
- R2. 每次自动运行后，系统必须生成一份完整报告，内容基于“真实订阅结果经过当前推荐算法筛选后的结果”，而不是原始抓取全集。
- R3. 系统必须把完整报告通过邮件发送到 `lijiawei14@huawei.com`。
- R4. 当日没有命中推荐结果时，系统仍然必须发送邮件，明确说明“今日无推荐论文”，避免把“无结果”和“任务失败”混淆。
- R5. 每封邮件必须包含统计信息，至少让用户能快速理解本次抓取规模、推荐命中情况和结果摘要。
- R6. 本地必须维护一个极简 HTML 页面，用于展示与邮件同版本的报告内容。
- R7. HTML 页面首版必须同时支持“最新报告展示”和“历史报告列表”，让用户可以查看最近一次结果并回看过去报告。
- R8. HTML 页面本次不实现点赞、纠错、确认和入库，但信息结构和页面布局必须允许后续平滑扩展这些交互能力，而不需要推翻首版产品形态。
- R9. 首版应坚持最小系统原则，优先打通每日运行、报告生成、邮件送达和本地查看四段主链路，不把用户反馈闭环纳入本次交付范围。

## Success Criteria

- 系统可以在被外部定时任务触发时，稳定完成一次真实 arXiv 订阅和推荐流程。
- 用户每天都能在邮箱中收到一封完整报告邮件；即使当天无推荐结果，也能收到清晰说明和统计信息。
- 本地 HTML 页面可以看到最新报告，并能浏览历史报告列表。
- 邮件内容与 HTML 页面展示内容保持一致，不出现“两套口径”。
- 首版交付后，后续要加入点赞、纠错、确认和入库能力时，可以在现有 HTML 页面上增量扩展，而不是重做产品入口。

## Scope Boundaries

- 本次不做点赞、纠错、确认和入库能力。
- 本次不做多用户产品化能力，默认只服务当前单一邮箱接收者和 Agent-first 场景。
- 本次不追求复杂运营分析、趋势看板或交互式仪表盘。
- 本次不要求把原始抓取结果作为主要用户交付物；用户主视角只看推荐后的结果。
- 本次不新增独立“recommend”产品入口，继续沿用现有业务命名边界。

## Key Decisions

- 定时调度不内建在代码里：每日北京时间 `08:00` 的触发由 `openclaw` 或 `codex` 定时任务承担，主仓代码只负责“单次被触发后的完整执行链路”。
- 邮件是第一交付物，HTML 是镜像和扩展壳子：这样能最快形成真实可用闭环，同时给后续反馈能力留宿主。
- 邮件直接发送完整报告：避免用户还要跳转本地页面查看核心内容，降低首版使用成本。
- 报告只展示推荐后的结果：首版目标是“把值得看的内容送达”，而不是“转发原始订阅数据”。
- 当天无推荐也发邮件：这样用户能确认任务正常运行，减少误判为任务失败。
- HTML 首版采用“最新报告 + 历史列表”：既满足最小可查看需求，也为后续扩展交互留出自然入口。

## Dependencies / Assumptions

- 默认当前推荐算法已经达到可上线状态，可直接用于 arXiv 订阅结果筛选。
- 默认系统具备可用的邮件发送通道，能够把自动生成的报告稳定发送到指定邮箱。
- 默认外部调度器会按北京时间 `08:00` 触发这条链路，主仓代码不负责实现定时调度本身。
- 默认自动任务的运行环境能够持续访问真实 arXiv 订阅来源。
- 默认本地 HTML 页面主要服务于运行机或维护者查看，不以公网部署为本次前提。

## Outstanding Questions

### Deferred to Planning

- [Affects R3][Technical] 邮件发送链路使用哪种具体发送方式，以及失败重试、告警和送达结果如何记录。
- [Affects R5][Needs research] 首版邮件统计信息最小集合应包含哪些字段，既足够有用又不引入过多实现负担。
- [Affects R6][Technical] HTML 页面应复用现有静态渲染能力到什么程度，如何与新的历史报告索引协作。
- [Affects R7][Technical] 历史报告的保留策略、排序规则和数量上限如何定义，才能兼顾最小实现和后续扩展。
- [Affects R8][Technical] 首版 HTML 需要预留哪些稳定标识或数据结构，才能让后续点赞、纠错、确认和入库扩展成本最低。
